# M3.2 — GS Frontend TUI

## Goal

Create `code/ground-station-frontend/` — a new lib+bin crate that is a pure REST
client of the GS backend. Library provides a trait-based backend client interface
and shared state model. The TUI binary (under `src/bin/`) renders live telemetry,
logs, and manual controls. Future binaries (e.g. web frontend) reuse the same lib.

**Architectural constraint** (per `docs/software/spec.md §5.4`): all frontends
never speak postcard-rpc. All data flows through the GS backend REST API.

## Library API (`src/lib.rs`)

The library exposes:

### BackendClient trait (`backend.rs`)

```rust
#[async_trait]
pub trait BackendClient: Send + Sync {
    /// GET /api/status — connection state + session summary.
    async fn status(&self) -> anyhow::Result<StatusResponse>;

    /// GET /api/records/latest — most recent telemetry record.
    async fn latest_record(&self) -> anyhow::Result<Option<Record>>;

    /// GET /api/logs?lines=N — recent GS-side log lines.
    async fn logs(&self, lines: usize) -> anyhow::Result<Vec<serde_json::Value>>;

    /// POST /api/commands/arm — arm the FC.
    async fn arm(&self) -> anyhow::Result<bool>;

    /// POST /api/commands/ignite — ignite the motor.
    async fn ignite(&self) -> anyhow::Result<bool>;
}

/// HTTP implementation using reqwest
pub struct HttpClientBackend {
    client: reqwest::Client,
    base_url: String,
}
```

| Method | Path | Response | Polling |
|---|---|---|---|
| GET | `/api/status` | `{connected, session_start, record_count}` | Every 500ms |
| GET | `/api/records/latest` | `Record` or 404 | Every 200ms |
| GET | `/api/logs?lines=N` | `Vec<JsonValue>` | Every 500ms |
| POST | `/api/commands/arm` | `{ok: true}` | On demand |
| POST | `/api/commands/ignite` | `{ok: true}` | On demand |
| POST | `/api/commands/ping` | `{latency_ms: number}` | Every `PING_EVERY_POLL` (default 1s) |

No explicit ping endpoint or button — connection health is shown by the status
bar's connected circle (●) blinking at the internal ping frequency. If the FC
responds, the circle stays solid/bright; on disconnect it turns red/off. This
keeps the UI clean and gives continuous heartbeat feedback without user action.

### Time-series history (`graph.rs`)

Ring buffer for each graphed metric, stored as timestamped samples. Velocity
and acceleration are **derived** from successive altitude readings (altitude
delta / time delta), since these aren't in the wire Record type.

```rust
/// Rolling window of time-ordered (time, value) pairs.
pub struct TimeSeries {
    samples: VecDeque<(Instant, f32)>,
    window: Duration,
}

impl TimeSeries {
    pub fn new(window: Duration) -> Self;
    /// Push a sample, evict samples outside the window.
    pub fn push(&mut self, time: Instant, value: f32);
    pub fn latest(&self) -> Option<f32>;
    pub fn len(&self) -> usize;
    pub fn min(&self) -> f32;
    pub fn max(&self) -> f32;
    /// Returns N evenly-spaced normalized [0, 1] values for rendering.
    pub fn sample(&self, columns: usize) -> Vec<f32>;
}
```

Velocity derivation: on each Altimeter record, compute `velocity = (alt - prev_alt) / dt`.
Acceleration: similarly from velocity deltas. Store prev_alt and prev_vel in the
poller closure.

### Shared state model (`state.rs`)

```rust
/// Captured when a FlightState transition record arrives — time + values at that instant.
#[derive(Clone, Debug)]
pub struct TransitionEvent {
    pub state: proto::flight_state::FlightState,
    pub time_since_start: f64,
    pub altitude_m: f32,
    pub velocity_ms: f32,
    pub acceleration_ms2: f32,
}

pub struct AppState {
    pub connected: ArcSwap<bool>,
    pub session_start: ArcSwap<Option<String>>,
    pub latest_record: ArcSwap<Option<Record>>,
    pub logs: ArcSwap<Vec<serde_json::Value>>,
    pub command_result: ArcSwap<Option<String>>,
    pub backend: Arc<dyn BackendClient>,
    // Time-series history (stores data points over configurable window)
    pub altitude_history: Mutex<TimeSeries>,
    pub velocity_history: Mutex<TimeSeries>,
    pub acceleration_history: Mutex<TimeSeries>,
    pub gps_history: Mutex<VecDeque<(Instant, GpsCoordinates)>>,
    // State transition log (captured at the instant each FlightState arrives)
    pub transitions: Mutex<Vec<TransitionEvent>>,
    /// Toggled by an internal heartbeat poller — the TUI renders the
    /// connected circle as blinking on/off at this rate.
    pub ping_heartbeat: ArcSwap<bool>,
}
```

Note: history fields use `Mutex<TimeSeries>` (not `ArcSwap`) because they are
updated frequently (every 200ms) and only read by the TUI render thread. A
simple `Mutex` is uncontended since the render thread holds the lock briefly to
snapshot the latest N samples. `ArcSwap` is kept for the last-known-value fields
where lock-free reads matter more.

### Response types (re-exported in `lib.rs`)

Wrappers for backend JSON responses (mirrors `routes.rs` types from GS backend).

- `main()` → init tracing, create state, spawn pollers, run TUI

pub fn to_chart_data(&self) -> Vec<(f64, f64)> {
    let start = self.samples.front().map(|(t, _)| *t).unwrap_or(Instant::now());
    self.samples.iter().map(|(t, v)| {
        let secs = t.duration_since(start).as_secs_f64();
        (secs, *v as f64)
    }).collect()
}

#### GPS trajectory — 2D trail with character-age markers

GPS positions are rendered as a 2D mini-trail showing the flight path over
the configurable time window (`GRAPH_TIME_WINDOW_S`, default 30s). Each
position fix is marked with a character indicating its age:

| Marker | Age | Meaning |
|---|---|---|
| `x` | 0–⅓ of window (≤10s at 30s window) | Current / recent |
| `o` | ⅓–⅔ of window (10s–20s) | Mid-aged |
| `.` | ⅔–end of window (20s–30s) | Oldest in window |

The trail is rendered as a single `Paragraph` widget spanning multiple rows,
with a fixed-width character grid. Each GPS fix is plotted as (col, row) by
normalizing lat/lon to the grid bounds, then the character is written at that
position with color intensity fading by age:

```
┌ GPS ─────────────── 30s window ────────────────┐
│                                                │
│       .                                        │
│     o  .                                       │
│   o    .                                       │
│  o     .                                       │
│ o       .                                      │
│x                                               │
│  ←─ latest x → o → . oldest ──→                │
│  Lat: 38.7320      38.7360      38.7400        │
│  Lon: -9.1370      -9.1340      -9.1310        │
└────────────────────────────────────────────────┘
```

Rendering approach:
- Normalize all lat/lon pairs in the window to a char grid (e.g. 20×6 chars)
- Overlay each fix: `x` for newest third, `o` for middle third, `.` for oldest third
- Color intensity: `x` = bright white, `o` = gray, `.` = dark gray
- If multiple fixes map to the same cell, the newer marker wins
- Below the trail: latest lat/lon + deltas for context

## Key architecture decisions

1. **Trait-based backend client** (`BackendClient` trait + `HttpClientBackend` impl) enables future frontends (e.g. web) to swap the transport (mock backend for testing, WebSocket, etc.).
2. **Polling state updaters** live in the library, not the binary — any frontend reuses them.
3. **`ArcSwap` for instant-last-known-value fields; `Mutex` for time-series history** — lock-free reads for the "latest" fields the TUI renders every frame; simple `Mutex` for history because it's uncontended (one async writer, one sync reader).
4. **`reqwest` with `rustls-tls`** — no OpenSSL dependency on Windows.
5. **Keybinding-based controls** — `a`=arm, `i`=ignite. Matches simulator's keyboard-first pattern.
6. **Velocity/acceleration derived from altitude deltas** — these aren't in the wire `Record` type, so they're computed client-side from successive altimeter readings.
7. **Chart widget with `Marker::Braille` + `GraphType::Line` for alt/vel/acc** — each data point (200ms) is a connected Braille dot. 3 charts stacked vertically, 6 rows each. State transition markers (`◆` deployment, `★` touchdown) overlaid at the exact time column.
8. **GPS 2D trail with age markers** — `x` (current), `o` (mid), `.` (old). Each marker age band is ⅓ of the configurable `GRAPH_TIME_WINDOW_S`. Rendered as a character grid on a `Paragraph` widget.
9. **Transition events captured at source** — when a `FlightState` record arrives, the telemetry poller snapshots current alt/vel/acc + timestamp. These are stored in `AppState::transitions` and displayed in a transitions table below the charts.
8. **No explicit ping command or display** — connection health is conveyed by the status bar's ● blinking at `PING_EVERY_POLL` interval. Solid/blinking = connected, red = disconnected.
9. **No deploy button** — deployment is entirely FC-controlled (FSM state `recovery_activated`). The frontend displays deployment state from `FlightState::RecoveryActivated` but never triggers it.
10. **`ArcSwap<bool>` for ping heartbeat** — a background poller toggles this at `PING_EVERY_POLL`; the TUI checkboxes it each frame to decide whether to render ● as green or dark.
