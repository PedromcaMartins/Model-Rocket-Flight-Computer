# M3.2 — GS Frontend TUI

## Goal

Create a ground-station frontend crate — a library that provides a generic
backend client interface and shared state model, plus a TUI binary that renders
live telemetry and manual controls. The crate is designed so future
frontends (e.g. a web UI) reuse the library without depending on the transport
layer.

**Architectural constraint** (per `docs/software/spec.md §5.4`): all frontends
never speak postcard-rpc. All data flows through the GS backend REST API.

## Scope

### In scope (M3.2)

- A **shared library** with:
  - Generic Backend client struct that follows `code/ground-station-backend/src/routes.rs`
  - Shared state model with per-field concurrency protection - multiple threads can use the shared state only have to synchronize when using the same attribute!
  - Rolling-window time-series history for telemetry metrics to display as charts - made generic to the type it is using (GPSCoordinates, f64, Velocity, ...)!
  - Background poller tasks (telemetry, status, heartbeat)
- A **TUI binary** using ratatui + crossterm that:
  - Captures and displays state-transition events with instant values
  - Provides keyboard-driven arm/ignite controls
  - Shows connection health via blinking heartbeat indicator
  - Follows the simulator's terminal lifecycle pattern
- **Backend** command placeholder endpoints — POST arm + ignite return OK
  - Post requests triggered by TUI should spawn threads to block on / await responses from gs-be, otherwise TUI render stops / slows down! 
  - Responses from gs-backend asynchronously update frontent shared state! 

### Out of scope (deferred)

- Arm/ignite wiring to the real FC → M3.3
- Simulator connection (sim-gs socket) → already in M3.1 - this is performed by gs-backend, not frontend!
- Web frontend binary → future milestone
- Deploy button → FC handles deployment autonomously; see design decision §6
- Ground station client-side does not derive data from fc-data!
- Renders alt/vel/acc time-series charts (Braille lines) + GPS 2D trail -> M3.4

## Architecture

### Library owns: transport abstraction, data model, polling lifecycle

The library (`lib.rs`) exposes everything any frontend needs, so the TUI binary
and future frontends share it unchanged:

- **Transport** — a `BackendClient` trait defining all backend
  operations the UI needs (status, telemetry, logs, commands). One HTTP / Websocket
  implementation ships with the crate; the trait exists so test doubles or
  alternative transports (WebSocket push) can be injected without touching
  the state layer or the UI code.
- **Shared state** — a single `AppState` holding all UI-visible data. Field-level
  concurrency protection is chosen per access pattern: `AtomicBool` for boolean
  flags, `ArcSwap` for latest-value structs/strings, `Mutex` for time-series
  history that's snapshot-read once per frame. See design decision §3.
- **Time-series history** — rolling-window ring buffer per metric (altitude,
  velocity, acceleration, GPS trail), parameterized by window duration and generic to the data type used.
- **Background pollers** — async functions spawned as `tokio::spawn` tasks with
  `CancellationToken`. Each data source (telemetry, status, heartbeat) has
  its own poller at its own rate. The telemetry poller also snapshots transition
  events when a `FlightState` record arrives. These poll functions have a generic structure:
  - receive AppState, clone gs-be client and cancellation token, tokio::select request and cancellation token, check request (where the logic is at), and wait for the next poll timer tick.
  - create a function that follows this pattern and received a function that actually is the logic of the program, removing unnecessary duplication!

### Binary owns: terminal lifecycle, render loop, input handling

The TUI binary (`src/bin/tui/`) depends on the library through its public API
only:

- Crossterm raw-mode setup/teardown, alternate screen, panic guard
- Render loop on a blocking thread at configurable FPS — snapshots `AppState`
  and composes the terminal layout each tick
- Keyboard event loop dispatching to actions (arm, ignite, quit, panel toggle)
- Mouse support will be implemented under M3.5
- No logic common to frontends shall be used under `src/bin`!

### Module boundaries

```
Library (lib) owns:
  BackendClient trait + impl  →  AppState + pollers  →  TimeSeries + metrics
                                    ↕ (public API)
Binary (tui) owns:
  terminal lifecycle  ↔  render loop (reads state)  ↔  input handler (calls backend)
```

Neither side depends on the other's internals. The binary accesses the backend
only through `AppState::backend: Arc<dyn BackendClient>`.

## REST API contract

The frontend polls these GS backend endpoints. Response shapes are defined in
the backend's route handlers — the frontend deserializes JSON into matching types.

| Method | Path | Response | Rate |
|---|---|---|---|
| GET | `/api/status` | `{connected, session_start, record_count}` | 500ms |
| POST | `/api/commands/arm` | `{confirmation}` - gs-be has confirmation = true placeholder | On demand |
| POST | `/api/commands/ignite` | `{confirmation}` - gs-be has confirmation = true placeholder | On demand |
| WEBSOCKET | `/api/records` | All records since start + continuous streaming | Continuous |

Arm and ignite are placeholder endpoints in M3.2 on gs-be — they return OK immediately.
For gs-fe, these endpoints request and receive from gs-be!
The actual FC and sim command wiring is deferred to M3.3.

An internal `/api/commands/ping` endpoint exists and is called by the heartbeat
poller (not the UI). Its result drives the ● blinking indicator and updates latency value on TUI — there is no
explicit ping button or latency display in the UI. See design decision §5.

## Key design decisions

Each decision records: the context that made it a decision, the choice taken,
alternatives considered, and why this path was chosen.

### 1. Abstract backend client (trait + impl)

**Context**: The frontend needs backend data, but transport details shouldn't
leak into the state layer or the UI.

**Decision**: Define a `BackendClient` trait with async methods. Ship one
`HttpClientBackend` implementation.

**Alternatives**: Hardcode reqwest calls into the state pollers. Use a concrete
type with generics.

**Rationale**: A trait lets future frontends (web via wasm) or test scenarios
provide their own implementation. The library code never imports reqwest directly.
Concrete generics would propagate transport choice through every consumer. A
trait keeps the coupling boundary clean.

### 2. Pushing and Polling with cancellation tokens

**Context**: Real-time telemetry could be push (WebSocket/SSE) or pull (polling).

**Decision**: Start with polling. Each data source has its own `tokio::spawn`
task with a `CancellationToken`.

**Alternatives**: Server-Sent Events from the backend. WebSocket for live feed.

**Rationale**: Polling is simpler to implement, debug, and test — the backend
already has the REST endpoints. Push can be added later by replacing poller
internals (or providing an alternative `BackendClient` impl) without touching
the state model or UI. Cancellation tokens give clean shutdown without shared
state.

### 3. Concurrency strategy: AtomicBool for flags, ArcSwap for values, Mutex for history

**Context**: The TUI render thread reads state at ~60Hz. Poller tasks write at
2-5Hz. Primitives should match each field's type and access pattern — a single
bool doesn't warrant an Arc's refcount overhead.

**Decision** (three tiers):

| Primitive | Targets | Reason |
|---|---|---|
| `AtomicBool` | `connected`, `ping_heartbeat` | Single CPU instruction load/store. No allocation, no refcount bump. The simplest possible synchronization for boolean flags. |
| `ArcSwap<T>` | `latest_record`, `command_result`, `session_start` | Lock-free reads on the hot render path. Writer clones the value and atomically swaps the pointer. Reader loads an `Arc` (refcount increment) — no blocking. |
| `Mutex<TimeSeries>` | altitude/velocity/acceleration/gps history, `transitions` | Uncontended: one async writer (5Hz), one sync reader (10Hz, brief snapshot). A simple `Mutex` is sufficient and simpler than alternatives. |

**Alternatives considered**:
- `RwLock` everywhere — same as Mutex in practice with a single reader; more overhead.
- `Mutex` for latest-value fields — adds jitter to the render loop; avoid on the hot path.
- `watch` channels — reader's `borrow()` blocks the writer for the borrow duration (up to
  ~16ms per frame at 60Hz), which can delay the 200ms telemetry poller.
- Whole-state `ArcSwap<AppState>` — gives consistent cross-field snapshots but forces a
  clone of every field's Arc on every write, even for a single-bool change.
- `ArcSwap<bool>` — works, but overkill. `AtomicBool` is a single instruction with the
  same semantics.

**Rationale**: The three-tier approach matches each field's size and access pattern to
the cheapest primitive that meets the requirements. Booleans are single bits — an atomic
load is one CPU instruction, no memory allocation, no refcount traffic. Complex types
need pointer indirection anyway, so ArcSwap's clone-and-swap cost is inevitable. History
collections are large, infrequently read, and uncontended — Mutex is the simplest correct
tool.

### 4. No explicit ping command or button

**Context**: Connection health feedback is needed, but a manual ping button adds
visual clutter and requires user action.

**Decision**: Omit a ping command entirely. The backend's ping endpoint is called
internally by a heartbeat poller that toggles `ping_heartbeat: AtomicBool`.
The TUI reads this each frame to render the status-bar ● as solid green when
connected, blinking when heartbeats arrive, red/dim on timeout.

**Alternatives**: Explicit ping button. Latency display in the status bar.
WebSocket keepalive.

**Rationale**: Continuous heartbeat feedback without user action is more useful
than a button the user must remember to press. It removes a visual element from
the already-dense layout. The blinking interval communicates "alive" without
requiring the user to read a number.

### 5. No deploy button

**Context**: Model rocket recovery is safety-critical — premature or missed
deployment destroys the rocket.

**Decision**: Do not provide a manual deploy command. Deployment is entirely FC-
controlled (`recovery_activated` state in the flight state machine).

**Alternatives**: Manual deploy button with confirmation. Software interlock.

**Rationale**: The FC's FSM has hard real-time constraints and redundant
detection (apogee + timer backup). A manual command from the ground station
bypasses these safeguards. The frontend displays deployment state from
`FlightState::RecoveryActivated` but never triggers it.

### 6. Extra info (defer)

**Context**: The ground-station would benefit from displaying
- Launchpad altitude
- Touchdown altitude
- Maximum values for altitude and acceleration

**Rationale**: The FC data is source-of-truth! ground-station cannot strapulate this data, because lost data packets can make it display this incorrect data!

### 7. Charts

Time-series charts should display a thick line with average values for that timeslot. if more that one value is used, also display (in thinner and dimmer color) the max and min values, making it 3 lines on a chart for a specific data type!

## Layout & UX

### Layout sketch (22 rows)

| Rows | Content |
|---|---|
| 1 | **Status bar**: connection ●, ping, session, record count |
| 2-7 | **Altitude chart**: 6-row Braille line, rolling 30s window |
| 8-13 | **Velocity chart**: same format |
| 14-19 | **Acceleration chart**: same format |
| 20 | **Transitions summary**: one row of markers + times + values |
| 21-22 | **GPS 2D trail** (left) + **Controls** (right): side by side |

Connection blinks with every confirmed ping response. If disconnected, status bar turns red!
The three Braille-line charts dominate the display. Each shows state-transition
markers at the exact column: one charactor for arming, `◆` for deployment, `★` for touchdown.

### GPS trail

Character-grid rendering of GPS positions over the configurable time window.
Each fix gets an age-fading character:

| Marker | Age band (30s window) | Color |
|---|---|---|
| `x` | 0-10s (newest) | Bright |
| `o` | 10-20s | Dim |
| `.` | 20-30s (oldest) | Dark |

### Controls

Keybinding display + last command result. Available action keys:

| Key | Action |
|---|---|
| `q` / Esc | Quit |
| `a` | Arm FC |
| `i` | Ignite motor |

### Disconnect UX

When the FC disconnects: red banner, dimmed data, "stale" badge on each chart,
Restart/Shutdown buttons at bottom.

## Patterns to follow

- **Config**: unit struct with `pub const` fields — see `flight-computer/src/config.rs`.
- **TUI lifecycle**: crossterm raw mode, alternate screen, `spawn_blocking` for
  render loop, scopeguard panic guard — follow the simulator's TUI setup exactly.
- **Error handling**: `anyhow::Result` for all fallible operations. No unwrap in
  production paths.
- **Logging**: `tracing` for both library and binary, matching workspace conventions.
- **Polling pattern**: `CancellationToken` for shutdown, configurable intervals
  via the Config struct, free functions that spawn `tokio::spawn` tasks. These live in the library, not the binary — any frontend reuses them.
- **`reqwest` with `rustls-tls`** — no OpenSSL dependency on Windows.
- **No explicit ping command or display** — connection health is conveyed by the status bar's ● blinking at `PING_EVERY_POLL` interval. Solid/blinking = connected, red = disconnected.
- **No deploy button** — deployment is entirely FC-controlled (FSM state `recovery_activated`). The frontend displays deployment state from `FlightState::RecoveryActivated` but never triggers it.
